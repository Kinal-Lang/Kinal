# Release Process

This document explains how to manage version numbers, prepare releases, and create Git tags.

---

## 1. Version Number Specification

Kinal uses [SemVer](https://semver.org/) three-part versioning: `MAJOR.MINOR.PATCH`

| Segment | When to Increment | Example Scenarios |
|---------|-------------------|-------------------|
| MAJOR | Breaking changes to the language or public API | Syntax overhaul, removal of existing features |
| MINOR | Backward-compatible new features | New syntax features, new standard library modules |
| PATCH | Backward-compatible bug fixes | Bug fixes, crash fixes |

> During development (MAJOR = 0): MINOR changes may also include breaking changes.

---

## 2. Dual-Version Strategy

The project maintains two independent version numbers, recorded together in the root `VERSION` file:

```
kinal=0.5.0
kinalvm=0.5.0
```

| Key | Component | Description |
|-----|-----------|-------------|
| `kinal` | Compiler + standard library | The primary version; most releases only need this updated |
| `kinalvm` | Virtual machine | VM-specific version, typically synchronized with kinal |

**Synchronization rules:**
- KinalVM's MAJOR and MINOR always match Kinal
- KinalVM's PATCH can be incremented independently, used only when the VM has fixes but the compiler does not

**Example:**
```
# When Kinal 0.6.0 is released, both are synchronized
kinal=0.6.0
kinalvm=0.6.0

# VM discovers an independent bug; only kinalvm line is changed
kinal=0.6.0
kinalvm=0.6.1
```

---

## 3. Changing Version Numbers

The **single source of truth** for version numbers is the root `VERSION` file. Do not edit `version.h` directly (it is auto-generated by CMake).

**Steps:**

1. Edit the root `VERSION` file, updating the relevant line(s) (typically both):
   ```
   kinal=0.6.0
   kinalvm=0.6.0
   ```
   If only the VM has an independent patch, change only the `kinalvm` line.
   ```bash
   python x.py dev
   ```

4. Verify the version was written correctly:
   ```bash
   kinal --version           # Output: Kinal 0.6.0
   kinal --version --verbose # Additional output: kinalvm 0.6.0
   kinalvm --version         # Output: KinalVM 0.6.0
   ```

---

## 4. Git Tag Format

| Scenario | Tag Format | Example |
|----------|------------|---------|
| Regular Kinal release | `v{VERSION}` | `v0.6.0` |
| KinalVM independent patch | `vm/v{VERSION}` | `vm/v0.6.1` |

```bash
# Create tag
git tag v0.6.0
git push origin v0.6.0
```

---

## 5. Release Checklist

Before tagging and releasing, confirm each of the following in order:

- [ ] All tests pass: `python x.py test`
- [ ] `VERSION` and `apps/kinalvm/VERSION` have been updated
- [ ] Rebuild succeeds: `python x.py dev`
- [ ] `kinal --version --verbose` outputs the correct version numbers
- [ ] Commit the version change and push
- [ ] Create and push the Git tag (see Section 4)
- [ ] Build the release package: `python x.py release`
- [ ] Verify the first line of `README.txt` inside the release package contains the correct version number
